# 任务 03：跨上下文运营 Dashboard

入口：`/backend-best-practices:cqrs-read-model-new`

## 任务输入

> 我们的系统已有订单（Ordering）、库存（Inventory）、客服工单（Support）三个模块，各自维护自己的数据库表。运营团队要一个统一 Dashboard：今日 GMV 与订单量趋势（小时粒度）、库存告急 SKU Top 20、未解决工单数按优先级分布。数据可以容忍 5 分钟延迟，但页面打开要快（< 1 秒）。运营按大区划分权限，只能看自己大区的数据。

## 判据（评审人持有，勿交给执行方）

### 应出现

| 判据 | 核对位置 |
| :-- | :-- |
| 先产出视图契约（三块视图各自的字段/聚合/筛选），fit 判定在视图之后 | 01-views → 02-fit-check 顺序 |
| 跨上下文聚合视图判 `use`（真实驱动：跨库 join 不可行 + 预聚合 + 延迟容忍）| fit 矩阵 |
| 每个字段标注来源上下文/事件/事实 | views + read-models |
| 同步策略与 5 分钟新鲜度匹配（定时刷新/CDC/事件投影任选，但有理由），含失败处理与重建 | sync 工件 |
| 大区权限下推进查询计划（非内存过滤），缓存键含大区维度 | 实现工件 |
| freshness 元数据（as_of/stale）出现在查询契约 | views/实现工件 |

### 不应出现

| 判据 | 核对位置 |
| :-- | :-- |
| 直接跨库 join 三个模块的表当作"简单方案" | fit / read-models |
| 把写侧 Domain 改造来伺候 Dashboard 字段 | decoupling 工件 |
| 双库/搜索引擎等超出需求的重型存储（除非给出真实驱动）| read-models |
| 查全量再按大区内存过滤 | 实现工件 |
