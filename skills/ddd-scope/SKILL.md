# DDD Scope（范围收敛）

发现阶段的第一步。它**不建模、不画事件流**，只做一件事：把一段"我想做个 XX 系统"的模糊愿望，逼问成一份**边界清晰、可被业务方确认、能直接喂给 `backend-best-practices:ddd-discover` 的建模输入**。它是整条 greenfield 链路的"地基浇筑"——地基歪了，后面所有建模都白做。

## 使用时机

- 用户提出一个全新系统/模块，但只有一句愿景式描述（"做个预订系统""搞个会员体系"）。
- `backend-best-practices:ddd-mode-router` 判定为 greenfield 且需求边界模糊，默认进入本 Skill。
- **被回溯触发**：下游 `backend-best-practices:ddd-subdomains` 无法区分核心域/支撑域（价值主张不清）、或 `backend-best-practices:ddd-discover` 发现问题边界自相矛盾时，回退到这里重划范围。

## 输入要求

- **必需**：用户的自然语言诉求（要做什么、给谁用、解决什么痛点）。
- **可选**：已有的业务文档、竞品、KPI/OKR、合规要求、团队与时限约束、`backend-best-practices:ddd-mode-router` 的路由决策卡。

## 流程

1. **抽取"问题陈述"**：用一句话写清"谁、在什么情境下、遇到什么问题、现在怎么凑合解决"。拒绝用"功能"代替"问题"（"要一个日历"是功能，"多人抢同一间会议室导致冲突"才是问题）。
2. **明确价值主张**：这个系统对组织/用户的**核心价值**是什么？为后续判定核心域埋下依据。
3. **划定目标 / 非目标**：列出本期要解决的（目标）与**明确不做的**（非目标）。非目标和目标同等重要——它是防止范围蔓延、也是 `backend-best-practices:ddd-contexts` 划边界的第一手依据。
4. **采集硬约束**：合规、SLA、与现有系统的集成约束、团队规模与技能、上线时限。区分"硬约束（不可协商）"与"偏好（可协商）"。
5. **播撒通用语言种子**：把用户口中反复出现的业务名词原样记下（不翻译成技术词），形成术语种子表，交给 `backend-best-practices:ddd-discover`/`backend-best-practices:ddd-contexts` 长成正式词汇表。
6. **登记风险与未知**：列出"现在还不知道、但会影响建模"的开放问题，标注阻塞级别。
7. **回放确认**：把上述结构化结果用业务语言复述给用户，得到确认后再放行。

## 输出

| 工件 | 结构要求 |
| :--- | :--- |
| 问题陈述 | 一句话：主体 + 情境 + 痛点 + 现状凑合方案 |
| 价值主张 | 1–3 条：系统为谁创造什么不可替代的价值 |
| 目标 / 非目标 | 两栏表：本期做（目标）/ 明确不做（非目标）各 3–7 条 |
| 硬约束清单 | 表格：约束项、类型（合规/集成/团队/时限）、是否可协商 |
| 通用语言种子 | 表格：业务术语、用户原话定义、疑似歧义标记 |
| 风险与开放问题 | 表格：问题、影响、阻塞级别（阻塞/重要/可延后） |

## 校验清单

- [ ] 问题陈述描述的是"问题"而非"功能"，且业务方能认同
- [ ] 价值主张能回答"为什么是我们做、为什么现在做"
- [ ] 非目标至少 3 条，且每条都堵住了一类可预见的范围蔓延
- [ ] 每条硬约束都标了"可否协商"，阻塞性约束已与用户对齐
- [ ] 术语种子全部来自用户原话，未被提前技术化
- [ ] 所有"阻塞级"开放问题已问清或已记录为显式假设

## 回溯触发

- 价值主张说不清、或目标与非目标互相矛盾 → 本 Skill 内部重做，不放行。
- 用户在回放确认时推翻问题陈述 → 重走流程 1–2。
- （上游极少触发）若发现这其实是改造既有系统 → 退回 `backend-best-practices:ddd-mode-router` 重新分流到 brownfield。

## 示例

```text
用户："我们想做个会议室预订系统，老是有人抢同一间房，还要能审批。"

→ backend-best-practices:ddd-scope 产出（节选）：

问题陈述：
  公司员工在临时安排会议时，因缺乏统一预订入口，多人对同一时段同一间会议室
  重复预订造成冲突；现状靠 IM 口头协调 + 门口贴纸，经常撞车。

价值主张：
  1. 杜绝时段冲突，让"订到 = 一定有房"。
  2. 高价值会议室（如董事会厅）的使用需要可审批、可追溯。

目标 / 非目标：
  目标：时段冲突检测；预订生命周期（申请/确认/取消）；高价值房审批流。
  非目标：访客签到、餐饮联动、跨公司租赁、设备报修（本期都不做）。

硬约束：
  | 约束 | 类型 | 可协商 |
  | SSO 必须接公司现有 LDAP | 集成 | 否 |
  | 8 周内上线试点 | 时限 | 可小幅协商 |

通用语言种子：
  预订(Booking)、时段(Slot)、会议室(Room)、审批(Approval)、冲突(Conflict)…

风险/开放问题：
  | 跨时区预订是否要支持？ | 影响时段建模 | 阻塞 |
  | 审批人按房间还是按部门定？ | 影响审批聚合 | 重要 |

→ 交给 backend-best-practices:ddd-discover 做事件风暴。
```
