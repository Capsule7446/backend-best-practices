# Scope（范围收敛）

## 做什么

把一段"我想做个 XX 系统"的模糊愿望，收敛成一份**边界清晰、可被业务方确认**的结构化范围输入。只做范围收敛——不建模、不画事件流。

## 需要什么参数

- **必需**：用户的自然语言诉求（要做什么、给谁用、解决什么痛点）。
- **可选**：已有业务文档、竞品、KPI/OKR、合规要求、团队与时限约束。

> 参数以工件文件形式在调用时给入；本能力不关心它由谁产生。

## 怎么做

1. **抽取问题陈述**：一句话写清"谁、在什么情境、遇到什么问题、现在怎么凑合解决"。拒绝用"功能"代替"问题"（"要一个日历"是功能，"多人抢同一间会议室导致冲突"才是问题）。
2. **明确价值主张**：这个系统对组织/用户的**核心、不可替代价值**是什么。
3. **划定目标 / 非目标**：本期要解决的（目标）与**明确不做的**（非目标）；非目标同等重要，用于堵住范围蔓延。
4. **采集硬约束**：合规、SLA、集成、团队规模、上线时限；区分"硬约束（不可协商）"与"偏好（可协商）"。
5. **播撒通用语言种子**：把用户口中反复出现的业务名词**原样**记下，不翻译成技术词。
6. **采集业务视图需求种子**：用户要"看什么"才能完成业务任务——每个视图记使用者、业务任务、要回答的问题、新鲜度容忍、权限预期。只记诉求，不设计读侧方案。
7. **登记风险与开放问题**：列出会影响后续、但现在还不确定的点，标注阻塞级别。
8. **回放确认**：用业务语言复述结构化结果，得到确认。

## 返回什么

一份**范围工件**，含下列各段：

| 段 | 结构 |
| :--- | :--- |
| 问题陈述 | 主体 + 情境 + 痛点 + 现状凑合方案 |
| 价值主张 | 1–3 条：为谁创造什么不可替代价值 |
| 目标 / 非目标 | 两栏各 3–7 条 |
| 硬约束 | 约束项、类型（合规/集成/团队/时限）、是否可协商 |
| 通用语言种子 | 业务术语、用户原话定义、疑似歧义标记 |
| 业务视图需求种子 | 每视图：view_id、使用者、业务任务、要回答的问题、latency/新鲜度容忍、read_your_writes 预期、权限种子 |
| 风险与开放问题 | 问题、影响、阻塞级别（阻塞/重要/可延后）|

> **返回格式自检**（仅查自己的产出格式）：七段齐全；问题陈述是"问题"非"功能"；非目标 ≥ 3 条；术语种子全部来自用户原话、未被提前技术化；视图需求种子只记诉求不带读侧方案；"阻塞级"开放问题已问清或记为显式假设。
