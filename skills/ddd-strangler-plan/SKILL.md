# DDD Strangler Plan（绞杀者迁移计划）

改造驱动的收尾规划站，也是 brownfield 第一纪律的承载者：**旧系统全程可运行、可回滚，绝不一次性重写。** 绞杀者模式（Strangler Fig）的精髓是让新实现像绞杀榕一样在旧系统外围逐步长出、逐步接管，直到旧实现自然枯萎。本 Skill 把接缝转化为**有序、可回滚的施工序列**：每个切片独立交付、先有特征化测试兜底、经 ACL 隔离、用流量切换灰度接管、随时可回滚。它产出的切片顺序驱动后段"逐切片复用 greenfield 战术建模"的循环。

## 使用时机

- `backend-best-practices:ddd-seam-finder` 已产出接缝清单、ACL 落点与切割风险评估。
- 需要把改造排成可执行、可回滚的迁移批次，再逐切片进入建模与落地。
- **被回溯触发**：逐切片落地中 `backend-best-practices:ddd-aggregates` 发现切片边界与聚合边界冲突时回到这里重排切片。

## 输入要求

- **必需**：`backend-best-practices:ddd-seam-finder` 的接缝清单、ACL 落点、切割风险评估、特征化测试需求；`backend-best-practices:ddd-code-survey` 的行为黑盒清单。
- **可选**：发布/灰度能力（特性开关、路由、影子流量）、回滚基础设施、业务低峰窗口。

## 流程

1. **切片化**：把改造目标按接缝切成**最小可独立交付**的切片（一个切片 = 一条缝 + 其 ACL + 一段可验收行为）。优先级 = 价值高 × 风险低 × 依赖少。
2. **排迁移批次**：按依赖与风险给切片定序——**先易后难、先低耦合后高耦合**，先拿低风险切片验证整套绞杀机制是否顺畅，再啃硬骨头。
3. **设计流量切换策略**：每片选灰度方式——特性开关、按比例路由、影子流量比对（新旧并行跑、只比对不接管）。明确"接管前需观察的指标"。
4. **设计回滚策略**：每片定义触发回滚的信号与一键回退动作，确保任何切片失败都能秒退回旧路径，数据不脏。
5. **圈定特征化测试范围**：按 `seam-finder` 与行为黑盒清单，明确每片切割前必须先补的特征化测试（锁住跨缝旧行为，作新实现的对照基线）。
6. **定义切片完成判据（DoD）**：特征化测试通过 + 新实现验收通过 + 流量已切且稳定 + 旧路径可下线/可回滚。
7. **规划旧代码退役**：每片接管稳定后，旧实现/旧字段/旧 ACL 的下线时机与清理动作（避免 ACL 永久留存成新债）。
8. **输出迁移路线图**：批次顺序、每片的切换/回滚/测试/DoD、整体里程碑。

## 输出

| 工件 | 结构要求 |
| :--- | :--- |
| 切片清单 | 表格：切片、对应接缝/ACL、可验收行为、价值/风险/依赖评分 |
| 迁移批次顺序 | 有序批次 + 排序理由（先易后难、依赖关系）|
| 流量切换策略 | 每片：灰度方式（开关/路由/影子）+ 接管前观察指标 |
| 回滚策略 | 每片：回滚触发信号 + 一键回退动作 + 数据安全保证 |
| 特征化测试范围 | 每片切割前必补的特征化测试清单 |
| 切片 DoD | 每片完成判据 |
| 旧代码退役计划 | 每片旧实现/旧字段/临时 ACL 的下线时机 |
| 迁移路线图 | 批次序列 + 里程碑总览 |

## 校验清单

- [ ] 每个切片可独立交付、独立回滚，不依赖未完成切片
- [ ] 批次顺序遵循"先易后难、先验证机制再啃硬骨头"
- [ ] 每片都有明确的流量切换方式与接管前观察指标
- [ ] 每片都有回滚触发信号与一键回退，且保证数据不脏
- [ ] 每片切割前的特征化测试范围已圈定（无裸切）
- [ ] 每片有可验收的 DoD
- [ ] 临时 ACL/旧字段都有退役时机，不留永久技术债
- [ ] 全程满足"旧系统始终可运行"的第一纪律

## 回溯触发

- 排序时发现切片之间无法解耦、无法独立交付 → 回退 `backend-best-practices:ddd-seam-finder`（接缝选错）。
- 某切片所需特征化测试覆盖不了旧行为 → 回退 `backend-best-practices:ddd-code-survey`（行为黑盒未解）。
- 逐切片落地时聚合边界与切片冲突 → 在本 Skill 重排切片（局部回溯）。

## 示例

```text
输入：订单单体的接缝（缝1履约/缝2计费/缝3物流读）+ ACL 落点 + 风险评估。

→ backend-best-practices:ddd-strangler-plan 产出（节选）：

切片清单：
  | S1 物流状态读隔离 | 缝3/视图 | 物流状态查询走新读模型 | 价值中/风险低/无依赖 |
  | S2 履约剥离 | 缝1/ACL-Fulfillment | 下单后发货走新履约上下文 | 价值高/风险中高/依赖S1 |
  | S3 计费剥离 | 缝2/ACL-Billing | 计费/退款走新计费上下文 | 价值高/风险高(含黑盒退款)/依赖S2 |

迁移批次顺序：S1 → S2 → S3
  理由：S1 低风险，先跑通绞杀+回滚机制；S2 中高风险但价值高；
        S3 含退款黑盒，留到机制成熟、特征化测试充分后再做。

流量切换：
  | S1 | 影子流量：新读模型与旧字段并行比对 7 天无差异再接管 |
  | S2 | 特性开关按 5%→50%→100% 灰度路由下单流量 |
  | S3 | 按支付渠道分批切，先小渠道 |

回滚策略：
  | S2 | 触发: 履约失败率↑或库存不一致告警 → 关开关秒回旧 OrderService；ACL 双写保证数据可对账 |

特征化测试范围：
  S2 前：锁"下单必扣库存、库存不足必拒单"现有行为
  S3 前：锁退款分账（行为黑盒）的现有输出快照

切片 DoD：特征化测试绿 + 新实现验收过 + 流量 100% 切稳 7 天 + 旧路径可回滚。

旧代码退役：
  S2 稳定后下线 OrderService.shipOrder 分支；全部完成后拆 Order.status 旧字段、下线临时 ACL。

迁移路线图：里程碑 M1(S1)→M2(S2)→M3(S3)，每里程碑独立可交付可回滚。

→ 进入逐切片循环：[补特征化测试] → backend-best-practices:ddd-aggregates(局部) → … → backend-best-practices:ddd-acceptance → 切流量 → 下一片。
```
