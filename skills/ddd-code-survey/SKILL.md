# Code Survey（逆向建模 / 代码勘测）

## 做什么

从"代码现实"出发，看懂一坨在跑、不能停的实现隐含的领域模型，产出"现状模型 + 坏味道清单 + 行为黑盒"。**先看清现状，不急着画理想模型**。

## 需要什么参数

- **必需**：可访问的代码库路径/仓库；改造目标（解耦/拆分/可测试性）。
- **可选**：数据库 schema、现有文档、关键流程口述、生产监控/日志。

## 怎么做

1. **定位业务核心代码**：区分"业务逻辑"与"技术管道"。
2. **逆推隐含聚合与实体**：哪些数据总是一起被改、被同一段逻辑保护，即隐含聚合候选。
3. **打捞散落的业务规则**：从 service/controller/存储过程/if 分支里捞出规则，还原成不变量候选。
4. **识别隐含上下文**：从命名漂移、模块边界、同名异义推断隐含限界上下文。
5. **列坏味道清单**：贫血模型、上帝类、原始类型偏执、事务脚本、跨模块直连库、把 DB 模型当领域模型等，每条标位置与影响。
6. **画现状模型**：用 DDD 词汇画系统**当前实际**（非理想）的模型，诚实反映耦合。
7. **现状 vs 目标差异**：与改造目标对照，列"差异即 backlog"。
8. **标行为黑盒**：还没看懂的旧行为，必须靠特征化测试兜底。

## 返回什么

| 工件 | 结构 |
| :--- | :--- |
| 现状领域模型 | 隐含聚合/实体/值对象草图 + 当前耦合（诚实版）|
| 隐含上下文清单 | 推断上下文、证据、混杂程度 |
| 业务规则打捞清单 | 规则、现所在位置、还原成的不变量候选 |
| 坏味道清单 | 类型、位置、影响、优先级线索 |
| 现状 vs 目标差异 | 差异点 -> 初步改造 backlog |
| 行为黑盒清单 | 尚未看懂、需特征化测试兜底的行为 |

> **返回格式自检**：现状模型解释的是**真实**行为而非理想；业务规则与技术管道已区分、规则尽量还原为不变量候选；隐含上下文推断都有代码证据；坏味道标了位置与影响；行为黑盒已显式登记、未"假装看懂"。
