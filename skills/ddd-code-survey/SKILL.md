# DDD Code Survey（逆向建模 / 代码勘测）

改造驱动的第一站，也是 brownfield 区别于 greenfield 的关键能力。greenfield 从"问题"出发，brownfield 必须先从"代码现实"出发——**你面对的是一坨已经在跑、不能停的实现，第一要务是看懂它隐含的领域模型，而不是急着画理想模型**。它产出的"现状模型 + 坏味道清单"是后续 `backend-best-practices:ddd-seam-finder` 找接缝、`backend-best-practices:ddd-strangler-plan` 排迁移的事实基础，也是与目标模型对照得出改造 backlog 的依据。`risk: caution`，因为它要读真实代码库。

## 使用时机

- `backend-best-practices:ddd-mode-router` 判定为 brownfield，准备改造既有系统。
- 需要在动任何代码前，先逆向出现状领域模型与问题地图。
- **被回溯触发**：`backend-best-practices:ddd-seam-finder` 发现现状模型不足以定位接缝、或 `backend-best-practices:ddd-acceptance` 的特征化测试无法覆盖旧行为（行为理解不足）时回到这里补勘测。

## 输入要求

- **必需**：可访问的代码库路径/仓库；改造目标（来自 router：解耦/拆分/可测试性）。
- **可选**：数据库 schema、现有文档、关键业务流程口述、生产监控/日志（看真实行为）。

## 流程

1. **定位业务核心代码**：先找出承载核心业务规则的模块（绕开框架/胶水代码），区分"业务逻辑"与"技术管道"。
2. **逆推隐含聚合与实体**：从类/表/函数反推——哪些数据总是一起被改、被同一段逻辑保护？这些就是隐含的聚合候选。识别隐含的实体、值对象、被贫血化的领域对象。
3. **打捞散落的业务规则**：业务规则常散落在 service、controller、存储过程、if 分支里。把它们捞出来，尽量还原成不变量候选（"订单已发货后不可改地址"）。
4. **识别隐含上下文**：从命名漂移、模块边界、团队归属、同名异义，推断系统里**隐含但未显性化的限界上下文**（订单里其实混着履约和计费两种语言）。
5. **列坏味道清单**：标记典型问题——贫血模型、上帝类、原始类型偏执（primitive obsession）、事务脚本、跨模块直连数据库、循环依赖、把 DB 模型当领域模型等，每条标位置与影响。
6. **画现状模型**：用 DDD 词汇画出系统**当前实际**的（而非理想的）模型与上下文草图，诚实反映耦合。
7. **现状 vs 目标差异**：把现状模型与改造目标对照，初步列出"差异即 backlog"，交给后续规划。
8. **标"行为黑盒"**：明确哪些旧行为还没完全看懂——这些必须靠特征化测试兜底，登记给 `backend-best-practices:ddd-strangler-plan`。

## 输出

| 工件 | 结构要求 |
| :--- | :--- |
| 现状领域模型 | 隐含聚合/实体/值对象草图 + 当前耦合关系（诚实版）|
| 隐含上下文清单 | 表格：推断出的上下文、证据（命名/边界/同名异义）、混杂程度 |
| 业务规则打捞清单 | 表格：规则、现所在位置（文件/函数/SP）、还原成的不变量候选 |
| 坏味道清单 | 表格：坏味道类型、位置、影响、改造优先级线索 |
| 现状 vs 目标差异 | 表格：差异点 → 初步改造 backlog 项 |
| 行为黑盒清单 | 尚未看懂、需特征化测试兜底的行为 |

## 校验清单

- [ ] 现状模型能解释系统**现有关键行为**（不是理想模型，是真实模型）
- [ ] 业务规则与技术管道已区分，规则已尽量还原为不变量候选
- [ ] 隐含上下文的推断都有代码证据（命名/边界/同名异义）
- [ ] 坏味道清单标了位置与影响，可指导后续接缝识别
- [ ] 行为黑盒已显式登记，未"假装看懂"
- [ ] 现状与改造目标的差异已转成初步 backlog

## 回溯触发

- 勘测发现这其实更适合重写而非改造（核心已不可救）→ 回退 `backend-best-practices:ddd-mode-router` 重新分流。
- 改造目标本身在代码现实下不成立（如"拆分"但数据强耦合到无法切）→ 回退 router 与用户重定目标。

## 示例

```text
输入：单体订单系统 src/order-monolith，目标=拆分出履约与计费。

→ backend-best-practices:ddd-code-survey 产出（节选）：

现状领域模型（诚实版）：
  一个上帝类 OrderService（3000+ 行）同时做：下单、扣库存、计费、开票、发货、退款。
  Order 表 30+ 列，混了下单信息、支付状态、物流单号、发票抬头。贫血 Order 对象只有 getter/setter。

隐含上下文清单：
  | 隐含上下文 | 证据 | 混杂度 |
  | 销售/下单 | createOrder、价格快照 | 中 |
  | 履约 | shipOrder、物流单号字段、库存扣减 | 高（与下单混在 OrderService）|
  | 计费 | charge、invoice、refund、发票字段 | 高（同上）|
  同名异义："Order.status" 在履约语境=物流状态，在计费语境=支付状态，被塞进同一字段！

业务规则打捞：
  | 已发货订单不可改收货地址 | OrderService.updateAddress() 第 1200 行的 if | 不变量候选 INV-A |
  | 已开票不可取消，只能退款 | 散落在 cancel() 与 refund() | INV-B |

坏味道清单：
  | 上帝类 OrderService | order/OrderService | 改一处牵全身 | 高 |
  | DB 模型当领域模型 | 全局 | 无法独立演进 | 高 |
  | 原始类型偏执（金额用 double）| 计费相关 | 精度/币种风险 | 中 |

现状 vs 目标差异（→backlog）：
  - 履约逻辑从 OrderService 剥离 → 独立履约上下文
  - 计费逻辑剥离 → 独立计费上下文
  - 拆分 Order.status 同名异义字段

行为黑盒：
  退款分账逻辑（涉及第三方支付回调）尚未完全看懂 → 标记需特征化测试。

→ 交给 backend-best-practices:ddd-seam-finder 定位接缝与 ACL 落点。
```
