# Contexts（限界上下文 + 通用语言）

## 做什么

把子域落成"解空间"的边界——限界上下文。在一条边界内每个术语只有一个明确含义、模型自洽。斩断"通用语言污染"：一个上下文 = 一套通用语言 = 一个可独立演进的模型。

## 需要什么参数

- **必需**：一份含 {子域分类, 核心域声明, 热点, 通用语言种子} 的工件。
- **可选**：组织架构（康威线索）、现有系统边界、团队划分。

## 怎么做

1. **子域映射上下文**：以子域为起点（不必 1:1），核心子域常对应一个核心上下文。
2. **用语言一致性切边界**：同一术语开始变味处即边界（"预订"在排期=时段占用，在计费=一笔费用）。**语言断裂线 = 上下文边界**。
3. **每上下文建独立词汇表**：同名异义术语在各自上下文分别定义，并标出它在别处对应概念。
4. **判定上下文性质**：标注承载的子域类型（core/supporting/generic），继承投入等级。
5. **写边界 ADR**：每条非显然边界写一条（背景/决策/被否方案/后果）。
6. **对齐组织（康威）**：检查边界是否与团队归属冲突。
7. **登记跨界关系待解**：把"上下文之间要交换什么"列为待办。

## 返回什么

| 工件 | 结构 |
| :--- | :--- |
| 上下文目录 | 上下文名、承载子域、性质、职责、负责团队 |
| 通用语言词汇表 | 每上下文一张：术语、本上下文定义、他处对应概念、歧义来源 |
| 边界 ADR | 背景、决策、被否选项、后果 |
| 跨界关系待解清单 | 上下文 A ↔ B、需交换的概念、方向 |

> **返回格式自检**：每上下文内术语唯一含义；同名异义都已被边界切开并分别定义；每条非显然边界都有 ADR（含被否方案）；上下文性质与子域分类一致；跨界要交换的概念已登记。
