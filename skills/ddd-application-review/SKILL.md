# Application Review（应用层审查）

## 做什么

对应用层做客观体检，设计态与实现态各评一次。只产出**证据与发现**；评分、放行与回溯由统筹方门禁负责（两级自检的第②级）。

## 需要什么参数

- **必需**：应用用例目录、编排设计；实现态另需应用层代码路径与测试清单。
- **可选**：流程管理器设计、关键用例名单（用于区分关键/一般覆盖率）。

## 怎么做

按八个维度逐项核对，每项记录 `适用数 / 满足数 / 证据`：

1. **用例完整性**：写命令↔UC 唯一映射；`Actor→Command→Handler→Domain→Result/Event→AC` 追踪；失败语义声明。
2. **分层职责**：Handler 内业务判断（`if status==…`、金额/资格计算）；直接依赖 ORM/DB/消息 SDK；契约暴露领域/ORM 实体；入口绕过应用层直连仓储。
3. **编排与领域委派**：步骤序声明；状态变更是否全部委派聚合/领域服务；每用例唯一主 Handler；同步/异步与失败终点明确。
4. **事务与并发**：写用例 UoW 覆盖；回滚声明；单聚合事务合规；并发敏感用例的版本/冲突语义；外部 I/O 在事务外或经 Outbox。
5. **可靠性**：可重试命令幂等覆盖（含崩溃窗口）；消费者去重；可靠事件 Outbox/等价；重试边界；长流程恢复与补偿。
6. **授权与数据边界**：受保护用例的用例级授权；tenant/owner 约束；身份来自可信安全上下文；敏感字段过滤。
7. **测试证据**（实现态）：每条义务给 `test_file + test_name + status`；只有"已覆盖"没有测试名视为**无证据**。负例必查：未认证、越权、跨租户、重复请求、并发冲突、回滚。
8. **硬约束扫描**：对硬约束清单逐条检查（关键用例无唯一 Handler；入口直连仓储/直改聚合；Domain 反向依赖；Application 依赖具体 SDK；关键不变量只在应用层检查；写用例无 UoW/回滚；无解释的跨聚合单事务；已提交事件可丢失；重复副作用无幂等；关键用例缺授权/存在跨租户路径；无界事务内外部调用；关键测试缺失或失败；长流程崩溃不可恢复），命中即单列。

## 返回什么

| 工件 | 结构 |
| :--- | :--- |
| 检查项证据表 | 维度 → 检查项 → 适用数/满足数/证据（文件/字段/测试名）；`N/A` 必须给理由 |
| 分级发现清单 | severity（critical/high/medium/low）+ area + problem + evidence + recommendation |
| 硬约束触发清单 | 命中的硬约束编号 + 证据（可为空） |
| 结论 | 事实性小结：各维度满足比例 + 关键缺口，不含放行判定 |

> **返回格式自检**：每条发现有证据可回查；每个比率有分子分母；`N/A` 有理由且不进分母；实现态每条测试义务有 test_file+test_name；不输出总分、不判定 pass/fail、不指定回溯目标（归统筹方）。
