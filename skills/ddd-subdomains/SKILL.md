# DDD Subdomains（子域分类）

战略阶段第一步。它回答一个投资问题而非技术问题：**哪块业务能力值得我们倾注最强的人和最精的模型（核心域），哪块够用就行（支撑域），哪块直接买现成（通用域）。** 分类错了，会把宝贵建模精力浪费在通用域、却草草对待真正的差异化能力。

## 使用时机

- `backend-best-practices:ddd-discover` 已产出事件流、泳道切分与能力候选。
- 需要在划限界上下文前，先想清楚"哪里是主战场"。
- **被回溯触发**：`backend-best-practices:ddd-contexts` 发现某上下文跨了核心与通用两种性质、或 `backend-best-practices:ddd-model-review` 判定精力错配时回到这里重分类。

## 输入要求

- **必需**：`backend-best-practices:ddd-discover` 的事件流、泳道切分；`backend-best-practices:ddd-scope` 的价值主张。
- **可选**：竞争分析、组织的战略目标、现有可采购的成熟方案清单。

## 流程

1. **抽取业务能力清单**：把事件流的每个泳道/职责块提炼为一项"业务能力"（动宾短语，如"检测时段冲突""审批高价值预订"）。
2. **逐项三分类**：
   - **核心域（Core）**：直接承载价值主张、构成差异化竞争力、外面买不到也不该买。
   - **支撑域（Supporting）**：业务必需但非差异化，需定制但不必最优。
   - **通用域（Generic）**：所有公司都一样、有成熟方案，应买/用开源（鉴权、通知、对账）。
3. **用两条检验校准**：①"如果这块做到行业最好，能带来竞争优势吗？"是→核心；②"市面上有没有现成的能直接用？"有且够用→通用。
4. **显式声明核心域**：核心域只能有少数几个；若超过 2–3 个，多半是没想清楚价值主张，需要再收敛。
5. **标注买/建/定制策略**：给每个子域一个落地策略（自研深耕 / 轻定制 / 采购集成）。
6. **回放与价值主张对齐**：核心域必须和 `backend-best-practices:ddd-scope` 的价值主张一一呼应，否则回溯。

## 输出

| 工件 | 结构要求 |
| :--- | :--- |
| 业务能力清单 | 列表：能力名（动宾）+ 一句话职责 + 来源泳道 |
| 子域分类表 | 表格：子域、类型（Core/Supporting/Generic）、分类理由、买建策略 |
| 核心域声明 | 1–3 个核心域 + 每个对应的价值主张条目 |
| 投入建议 | 表格：子域 → 投入等级（高/中/低）→ 人/模型精度预期 |

## 校验清单

- [ ] 每项能力都被唯一归类，无"既核心又通用"的模糊项
- [ ] 核心域数量 ≤ 3，且每个都映射到一条价值主张
- [ ] 每个通用域都给出了"可采购/可复用"的现成方案方向
- [ ] 买/建策略与硬约束（团队规模、时限）自洽
- [ ] 分类理由用业务语言写明，而非"因为它复杂"

## 回溯触发

- 找不到任何核心域，或核心域 > 3 且无法收敛 → 回退 `backend-best-practices:ddd-scope`（价值主张不清）。
- 某能力无法归类，因为事件流缺失关键环节 → 回退 `backend-best-practices:ddd-discover` 补风暴。

## 示例

```text
输入：backend-best-practices:ddd-discover 的泳道切分 [预订核心][审批][通知][身份] + 价值主张。

→ backend-best-practices:ddd-subdomains 产出（节选）：

业务能力清单：
  - 检测时段冲突（预订核心）
  - 管理预订生命周期（预订核心）
  - 审批高价值预订（审批）
  - 发送预订通知（通知）
  - 鉴别员工身份（身份）

子域分类表：
  | 子域 | 类型 | 理由 | 策略 |
  | 预订与冲突 | Core | 价值主张"订到=有房"全靠它；竞品做不到细粒度冲突检测 | 自研深耕 |
  | 审批 | Supporting | 业务需要但规则常变，非差异化 | 轻定制（可配置规则）|
  | 通知 | Generic | 邮件/IM 推送人人都一样 | 采购/接现有通知服务 |
  | 身份 | Generic | 接公司 LDAP（硬约束）| 集成现成 |

核心域声明：
  «预订与冲突» —— 对应价值主张 1「订到=一定有房」。

投入建议：
  | 预订与冲突 | 高 | 最强建模、最严不变量 |
  | 审批 | 中 | 够用即可，留可配置空间 |
  | 通知/身份 | 低 | 防腐层包一层即可 |

→ 交给 backend-best-practices:ddd-contexts 划限界上下文与通用语言。
```
