# Acceptance（面向接口的验收）

## 做什么

验"实现是否忠于契约、不变量在各路径下是否恒真"，而非"代码跑没跑"。验证锚定在**语言中立的契约**上，故同一套验收意图可在任何语言复用。

## 需要什么参数

- **必需**：不变量命题集与 Given-When-Then 验收准则；实现与依赖装配。
- **可选**：特征化测试（characterization test，锁旧行为的对照基线）；性能/并发验收要求。

## 怎么做

1. **不变量测试**：每条 INV 写正例（保持为真）+ 反例（非法被拒），尤其测边界与并发竞态。
2. **端口契约测试**：与实现无关，验前置/后置、查询边界、失败/空值语义；同一契约测试可套在该端口任何适配器上。
3. **用例验收测试**：把 Given-When-Then 逐条落成端到端用例，走应用服务编排路径。
4. **事件与最终一致验收**：在故障注入下验证事件/订阅/补偿仍收敛到一致。
5. **依赖方向回归**：加架构测试/静态检查，断言领域核心不依赖基础设施。
6. **（有特征化测试时）行为等价验收**：对照新旧实现确认对外可观测行为一致。
7. **汇总验收报告**：每条 INV/AC 的覆盖与通过状态 + 结论。

## 返回什么

| 工件 | 结构 |
| :--- | :--- |
| 不变量测试集 | 每条 INV：正例+反例+边界/并发，标 INV 编号 |
| 端口契约测试 | 与实现无关；可复用于该端口任何适配器 |
| 用例验收测试 | 由 Given-When-Then 一一落成 |
| 最终一致验收 | 事件/补偿在故障注入下的收敛验证 |
| 架构依赖测试 | 断言领域核心零基础设施依赖 |
| 验收报告 | 每条 INV/AC 覆盖与通过状态 + 结论 |

> **返回格式自检**：每条不变量都有正例+反例（含关键并发）；每个端口有与实现无关的契约测试；每条 Given-When-Then 都有对应用例测试；最终一致路径在故障注入下验证收敛；架构依赖测试存在；报告对每条 INV/AC 给出明确通过/阻断结论。
