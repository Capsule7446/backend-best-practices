# Acceptance（面向接口的验收）

## 做什么

验"实现是否忠于契约、不变量在各路径下是否恒真"，而非"代码跑没跑"。验证锚定在**语言中立的契约**上，故同一套验收意图可在任何语言复用。

## 需要什么参数

- **必需**：不变量命题集与 Given-When-Then 验收准则；应用用例契约（Input Port、错误语义、可靠性语义）；实现与依赖装配。
- **可选**：特征化测试（characterization test，锁旧行为的对照基线）；性能/并发验收要求。

## 怎么做

1. **不变量测试**：每条 INV 写正例（保持为真）+ 反例（非法被拒），尤其测边界与并发竞态。
2. **端口契约测试**：与实现无关，验前置/后置、查询边界、失败/空值语义；同一契约测试套在该端口所有实现上（In-memory 与真实适配器同套）。
3. **用例验收测试**：把 Given-When-Then 逐条落成端到端用例，从 Input Port 进入走应用编排路径——不绕过 Handler 直调聚合。
4. **应用编排测试**：用 Fake/In-memory 端口验证编排本身——授权通过与拒绝、加载/保存顺序、领域拒绝时不保存、错误映射；不重复测领域规则。
5. **事务与幂等验收**：聚合保存与 Outbox 同提交/同回滚；领域拒绝无任何写入；并发版本冲突不覆盖写；重复请求返回原结果、同 key 异载荷被拒。
6. **事件与最终一致验收**：在故障注入下验证事件/订阅/补偿仍收敛到一致（重复、乱序、重放）。
7. **依赖方向回归**：架构测试断言——领域核心不依赖应用/基础设施；应用不依赖具体 SDK；入站适配器不直连仓储。
8. **（有特征化测试时）行为等价验收**：对照新旧实现确认对外可观测行为一致。
9. **汇总验收报告**：每条 INV/AC/测试义务的覆盖与通过状态（含 test_file+test_name 证据）+ 结论。

## 返回什么

| 工件 | 结构 |
| :--- | :--- |
| 不变量测试集 | 每条 INV：正例+反例+边界/并发，标 INV 编号 |
| 端口契约测试 | 与实现无关；可复用于该端口任何适配器 |
| 用例验收测试 | 由 Given-When-Then 一一落成，从 Input Port 进入 |
| 应用编排测试 | Fake 端口下的编排验证：授权/顺序/拒绝不保存/错误映射 |
| 事务与幂等验收 | 同提交同回滚、拒绝零写入、并发冲突、幂等重复/异载荷 |
| 最终一致验收 | 事件/补偿在故障注入下的收敛验证 |
| 架构依赖测试 | 领域零基础设施依赖；应用零具体 SDK；入口不直连仓储 |
| 验收报告 | 每条 INV/AC/义务覆盖与通过状态（test_file+test_name）+ 结论 |

> **返回格式自检**：每条不变量都有正例+反例（含关键并发）；每个端口有与实现无关的契约测试；每条 Given-When-Then 都有对应用例测试且经 Input Port 进入；写用例有事务/幂等验收；最终一致路径在故障注入下验证收敛；架构依赖测试覆盖三条禁令；报告每项有测试名证据并给出明确通过/阻断结论。
