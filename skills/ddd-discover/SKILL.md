# Discover（领域发现 / 事件风暴）

## 做什么

用事件风暴把抽象的问题陈述，变成一条按时间排序、由业务事件驱动的领域流，并暴露真正难的地方（热点）。

## 需要什么参数

- **必需**：一份含 {问题陈述, 目标/非目标, 通用语言种子} 的工件。
- **可选**：领域专家问答渠道、既有流程文档、用户旅程图。

## 怎么做

1. **铺领域事件**：用"已发生的事"口吻按时间排出关键节点，一律**过去式 + 业务语言**（"预订已确认"而非"insert 一行"）。先求全。
2. **补命令与参与者**：每个事件由什么命令触发、谁发的，配对"命令→事件"。
3. **标策略与外部系统**：哪些由规则自动触发后续（策略）、哪些依赖外部系统。
4. **圈热点**：凡"看情况/有争议/偶尔"处一律标红——热点是模型最难也最有价值处，不抹平。
5. **聚候选数据**：每个命令/事件涉及的关键业务数据（只记概念，不画字段）。
6. **切泳道**：按时间/职责断点把长事件流切段，给分界留线索。
7. **整理歧义清单**：把热点整理成带问题的清单，能问的问、问不到的记为显式假设。

## 返回什么

| 工件 | 结构 |
| :--- | :--- |
| 领域事件流 | 时间排序：事件（过去式）+ 触发命令 + 参与者 |
| 命令/事件目录 | 命令 → 产生事件 → 关键数据 |
| 策略清单 | 触发事件 → 规则 → 后续命令/事件 |
| 外部依赖清单 | 交互点 → 外部系统 → 方向 |
| 热点清单 | 热点、所在事件、争议点、是否已澄清 |
| 泳道切分 | 分段 + 每段职责标签 |

> **返回格式自检**：事件全为"过去式 + 业务语言"、无 CRUD 词；每个事件可追溯到命令与参与者；争议点都已标热点；策略与命令已区分；泳道段都有职责标签。
