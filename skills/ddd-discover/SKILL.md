# DDD Discover（领域发现 / 事件风暴）

发现阶段的第二步，也是整套体系里**最发散、最依赖业务知识**的一环。它用事件风暴（Event Storming）的方法，把抽象的问题陈述变成一条**按时间排序、由业务事件驱动**的领域流，并暴露出真正难的地方（热点）。它产出的事件流是后续切子域、划上下文、定聚合的共同事实基础。

## 使用时机

- `backend-best-practices:ddd-scope` 已产出问题陈述、目标与术语种子。
- 需要把"领域里发生的事"显性化，作为战略建模的输入。
- **被回溯触发**：`backend-best-practices:ddd-aggregates` 发现某不变量缺少触发事件、或 `backend-best-practices:ddd-subdomains` 发现能力清单不完整时，回到这里补风暴。

## 输入要求

- **必需**：`backend-best-practices:ddd-scope` 的问题陈述、目标/非目标、通用语言种子。
- **可选**：领域专家可问答的渠道、既有流程文档、用户旅程图。

## 流程

1. **铺领域事件**（橙）：让业务专家用"已发生的事"口吻，按时间顺序说出关键节点——一律用**过去式 + 业务语言**命名（"预订已确认"而非"insert 一行"）。先求全，不求精。
2. **补命令与参与者**（蓝/黄）：每个事件前面是什么**命令**触发的？谁（参与者/角色）发的？把"命令 → 事件"配对。
3. **标策略与外部系统**（紫/粉）：哪些事件由规则自动触发后续动作（策略："只要时段冲突就拒绝"）？哪些依赖外部系统（LDAP、邮件）？
4. **圈热点**（红）：凡是出现"看情况""要讨论""这块有争议""偶尔会……"的地方，统统贴红点——**热点 = 模型最难也最有价值的地方**，绝不抹平。
5. **聚拢候选数据**（绿）：每个命令/事件需要或产生哪些关键业务数据，作为后续值对象/聚合属性的线索（不画字段，只记概念）。
6. **切泳道**：按明显的时间/职责断点把长事件流切成几段，为 `backend-best-practices:ddd-subdomains`/`backend-best-practices:ddd-contexts` 预留分界线索。
7. **整理歧义清单**：把所有红点热点整理成带问题的清单，能当场问业务专家的就问，问不到的标为开放假设。

## 输出

| 工件 | 结构要求 |
| :--- | :--- |
| 领域事件流 | 时间排序列表：事件（过去式）+ 触发命令 + 参与者 |
| 命令/事件目录 | 表格：命令 → 产生事件 → 关键数据（绿） |
| 策略清单 | 表格：触发事件 → 规则 → 后续命令/事件 |
| 外部依赖清单 | 表格：交互点 → 外部系统 → 方向（入/出） |
| 热点清单 | 表格：热点描述、所在事件、争议点、是否已澄清 |
| 泳道切分 | 初步分段及每段的职责标签（子域候选） |

## 校验清单

- [ ] 事件全部用"过去式 + 业务语言"，无技术/CRUD 词汇
- [ ] 每个领域事件都能追溯到触发它的命令与参与者
- [ ] 所有"看情况/有争议/偶尔"的点都已标为热点，未被抹平
- [ ] 策略（自动规则）与命令（人/系统主动触发）已区分清楚
- [ ] 热点清单中阻塞级歧义已问清或记为显式假设
- [ ] 事件流覆盖了 `backend-best-practices:ddd-scope` 的全部目标，未遗漏价值主张相关路径

## 回溯触发

- 事件流揭示问题边界与 `backend-best-practices:ddd-scope` 的目标/非目标矛盾 → 回退 `backend-best-practices:ddd-scope` 重划范围。
- 热点过多且互相纠缠、说明价值主张本身没想清 → 回退 `backend-best-practices:ddd-scope` 重述价值。

## 示例

```text
输入：backend-best-practices:ddd-scope 的会议室预订问题陈述 + 术语种子。

→ backend-best-practices:ddd-discover 产出（节选）：

领域事件流（时间序）：
  [发起预订] —员工→ 预订已提交
  → （策略：时段冲突？）→ 冲突已检出 / 时段已锁定
  → 预订已确认（普通房） | 预订待审批（高价值房）
  → 审批已通过 / 审批已驳回
  → 预订已生效 → （会议日）→ 预订已结束
  旁路：预订已取消 → 时段已释放

命令/事件目录：
  | 命令 提交预订 | → 预订已提交 | 数据：申请人/房/时段 |
  | 命令 审批预订 | → 审批已通过/驳回 | 数据：审批人/意见 |

策略清单：
  | 预订已提交 → 若房为"高价值" → 需走审批 → 预订待审批 |
  | 预订已提交 → 若时段已占 → 冲突已检出（直接拒绝）|

外部依赖：
  | 登录鉴权 → LDAP（入）| 确认通知 → 邮件服务（出）|

热点（红）：
  | 跨时区时段如何统一表达？ | 在"时段已锁定" | 阻塞 |
  | 审批中能否被他人抢占时段？（锁的粒度）| 在"预订待审批" | 重要 |
  | 取消已生效预订是否要审批回收？ | 在"预订已取消" | 可延后 |

泳道切分（子域候选）：
  [预订核心]  [审批]  [通知]  [身份]

→ 交给 backend-best-practices:ddd-subdomains 做核心/支撑/通用分类。
```
