# DDD Seam Finder（接缝与防腐层识别）

改造驱动第二站。逆向出现状模型后，改造的现实问题是：**这坨耦合代码，到底能从哪儿安全地切开？** "接缝（seam）"是 Michael Feathers 的概念——能在不改动其行为的前提下改变其行为的地方。本 Skill 在现状模型上定位这些缝（天然的接口/事件/模块边界），并指定**防腐层（ACL）**的落点，把要剥离的新模型与残留旧代码隔开。它产出的是"在哪下刀、用什么隔离、风险多大"，直接喂给 `backend-best-practices:ddd-strangler-plan` 排迁移顺序。

## 使用时机

- `backend-best-practices:ddd-code-survey` 已产出现状模型、隐含上下文与坏味道清单。
- 需要确定改造的切入边界与防腐层位置，再排迁移计划。
- **被回溯触发**：`backend-best-practices:ddd-strangler-plan` 发现切片边界与可用接缝冲突、或落地时发现接缝处行为耦合超预期时回到这里。

## 输入要求

- **必需**：`backend-best-practices:ddd-code-survey` 的现状模型、隐含上下文清单、坏味道清单、行为黑盒清单。
- **可选**：调用关系/依赖图、数据库外键关系、运行时调用频次（评估切割风险）。

## 流程

1. **沿隐含上下文边界找缝**：`backend-best-practices:ddd-code-survey` 推断的隐含上下文之间，是天然的候选接缝（履约 vs 计费）。优先在这些语言断裂处下刀。
2. **盘点可用接缝类型**：识别每处可利用的接缝——已有接口/抽象、可插入接口的调用点、可改为事件的同步调用、模块/包边界、数据库视图层。
3. **定位防腐层落点**：在"新剥离模型"与"残留旧系统"之间，确定 ACL 的物理落点——新模型通过 ACL 读写旧系统/旧库，把旧的脏概念翻译成干净领域语言，**防止旧模型污染新模型**。
4. **评估每条缝的切割成本与风险**：数据耦合（共享表/外键）、事务耦合（跨边界事务）、调用频次、隐藏副作用。给每条缝打"易切/难切/暂不可切"。
5. **处理同名异义字段**：对 `code-survey` 标出的混用字段（如 Order.status 同时表履物流与支付），规划在接缝处拆分/翻译。
6. **标特征化测试需求**：每条缝在切割前需要哪些特征化测试锁住跨缝行为，登记给 `backend-best-practices:ddd-strangler-plan` 和 `backend-best-practices:ddd-acceptance`。
7. **输出接缝地图**：把所有缝、ACL 落点、风险等级画成一张可决策的地图。

## 输出

| 工件 | 结构要求 |
| :--- | :--- |
| 接缝清单 | 表格：缝位置、接缝类型（接口/事件/模块/视图）、连接的两侧、切割难度 |
| 防腐层落点 | 表格：ACL 名、隔离的新旧两侧、翻译的概念、读/写方向 |
| 切割风险评估 | 表格：每条缝的数据/事务/频次/副作用风险 + 综合等级 |
| 同名异义拆分方案 | 混用字段 → 在缝处如何拆分/翻译 |
| 特征化测试需求 | 每条缝切割前需锁住的跨缝行为清单 |
| 接缝地图 | 现状模型上标注所有缝、ACL、风险等级 |

## 校验清单

- [ ] 每个隐含上下文边界都评估过是否为可用接缝
- [ ] 每条要切的缝都指定了接缝类型与切割难度
- [ ] 新剥离模型与旧系统之间都规划了 ACL，方向明确
- [ ] 数据/事务耦合风险已逐缝评估，"暂不可切"项已标明原因
- [ ] 同名异义字段在缝处的拆分方案已给出
- [ ] 每条缝的特征化测试需求已登记
- [ ] 接缝选择服务于改造目标（解耦/拆分），非为切而切

## 回溯触发

- 找不到任何低风险接缝、处处强耦合 → 回退 `backend-best-practices:ddd-code-survey`（现状模型不够细，或需重估改造目标）。
- 拟定的 ACL 翻译不动旧模型（旧概念与新概念结构性冲突）→ 回退 `backend-best-practices:ddd-code-survey` 补理解，必要时 `backend-best-practices:ddd-mode-router` 重定目标。

## 示例

```text
输入：订单单体现状模型（OrderService 上帝类，履约/计费混杂，Order.status 同名异义）。

→ backend-best-practices:ddd-seam-finder 产出（节选）：

接缝清单：
  | 缝1 履约边界 | 事件接缝 | 下单 ↔ 履约 | 中：扣库存当前是同步调用，可改为 OrderPlaced 事件 |
  | 缝2 计费边界 | 接口接缝 | 下单 ↔ 计费 | 中：charge() 抽成 BillingPort 接口注入 |
  | 缝3 物流状态读 | 视图接缝 | 旧 Order 表 ↔ 新履约模型 | 易：先用读视图隔离物流字段 |

防腐层落点：
  | ACL-Fulfillment | 新履约模型 ↔ 旧 Order 表/库存 | 翻译"订单行→发货单"，读旧库写新模型 | 双向 |
  | ACL-Billing | 新计费模型 ↔ 旧支付回调 | 翻译第三方支付术语→领域语言 | 入站为主 |

切割风险：
  | 缝1 | 数据: 共享库存表(中) 事务: 下单扣库存当前同一事务(高) 频次: 高 | 综合: 中高，先补事务边界 |
  | 缝2 | 数据低 事务中 频次中 | 综合: 中 |

同名异义拆分：
  Order.status → 拆为 FulfillmentStatus(履约上下文) + PaymentStatus(计费上下文)，
  ACL 在缝处双写过渡，迁移完成后下线旧字段。

特征化测试需求：
  - 缝1：锁住"下单成功必扣库存"的现有行为
  - 计费：锁住退款分账（行为黑盒）现有输出，再动实现

接缝地图：以 OrderService 为中心，标 缝1/缝2/缝3 与两个 ACL 落点，风险色标。

→ 交给 backend-best-practices:ddd-strangler-plan 排迁移批次与回滚策略。
```
