# Read Model Patterns

## DTO

- 适用场景：查询形状接近写模型，只需要屏蔽 Domain Entity。
- 不适用场景：跨领域聚合、复杂统计、重搜索。
- 优点：最简单、低运维成本。
- 缺点：容易退化成 Entity 包装。
- 最小实现：Query Service 返回专用 Response DTO。
- 常见误用：DTO 字段反向驱动 Aggregate 设计。

## Query Service

- 适用场景：查询逻辑需要独立于命令用例，但仍可实时组合。
- 不适用场景：查询成本高到影响主库稳定。
- 优点：边界清楚，不强制新存储。
- 缺点：复杂查询会堆积。
- 最小实现：`*QueryService` + 查询 DTO + 显式权限检查。
- 常见误用：把业务写规则放进 Query Service。

## Database View

- 适用场景：同库 join/投影稳定，需要复用查询形状。
- 不适用场景：跨库、跨服务、需要复杂刷新控制。
- 优点：实现轻，字段来源可见。
- 缺点：性能依赖数据库优化。
- 最小实现：只读 View + 查询端 DTO。
- 常见误用：把 View 当作业务真源写入。

## Materialized View

- 适用场景：报表、dashboard、统计聚合，允许延迟。
- 不适用场景：用户必须立即看到写后结果。
- 优点：查询快，适合预计算。
- 缺点：刷新策略和旧数据提示必须清楚。
- 最小实现：物化视图 + 定时/按需刷新 + `generatedAt`。
- 常见误用：隐藏 stale data。

## Read Table

- 适用场景：读侧需要反范式化、稳定索引、独立查询契约。
- 不适用场景：简单 CRUD。
- 优点：面向视图优化，查询成本可控。
- 缺点：需要同步、重建和一致性策略。
- 最小实现：同库 read table + 同步更新或 job 刷新。
- 常见误用：Read Table 变成写侧真源。

## Cache

- 适用场景：热点查询、可重建、可容忍短暂旧数据。
- 不适用场景：强一致交易确认。
- 优点：低延迟。
- 缺点：失效策略容易复杂。
- 最小实现：缓存 key、TTL、失效/重建路径。
- 常见误用：缓存里保存不可恢复的业务事实。

## Search Document

- 适用场景：全文搜索、多条件筛选、排序打分。
- 不适用场景：简单列表。
- 优点：搜索能力强。
- 缺点：最终一致和重建成本。
- 最小实现：搜索索引文档 + 异步刷新 + rebuild job。
- 常见误用：为普通分页引入搜索引擎。

## Analytics Table

- 适用场景：BI、历史分析、大量聚合扫描。
- 不适用场景：在线交易读路径。
- 优点：适合大规模分析。
- 缺点：链路长、延迟高。
- 最小实现：ETL/ELT + 分区表 + 数据时间戳。
- 常见误用：让运营报表反向决定在线 Domain。
