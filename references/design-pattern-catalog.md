# Design Pattern Catalog

语言中立原则：先确认变化轴，再选择模式；先定义角色和协作，再落到语言实现。不要把 Java 类图当成所有语言的目标形态。

## 创建型模式

| 模式 | 意图 | 适用 | 避免 |
| :--- | :--- | :--- | :--- |
| Abstract Factory | 创建一组相关对象，隐藏具体产品族 | 同一业务有多套兼容产品族，如不同支付渠道/主题组件 | 产品族很少或不会成套变化 |
| Builder | 分步骤构建复杂对象，隔离构造过程 | 构造参数多、有校验、有默认值或多种表示 | 只是少量参数，可用普通构造/命名参数 |
| Factory Method | 延迟具体对象创建到子类/策略函数 | 框架定义流程，扩展方决定具体产品 | 创建逻辑没有扩展点 |
| Prototype | 通过复制现有实例创建新对象 | 创建成本高、运行时配置对象需要克隆 | 对象有复杂外部资源或身份语义 |
| Singleton | 保证单实例访问 | 进程级共享资源且生命周期可控 | 业务服务、可变全局状态、测试需隔离的对象 |

## 结构型模式

| 模式 | 意图 | 适用 | 避免 |
| :--- | :--- | :--- | :--- |
| Adapter | 转换接口，让不兼容对象协作 | 接入第三方 API、遗留接口、防腐层 | 双方接口都可直接修改 |
| Bridge | 分离抽象与实现，使二者独立变化 | 维度组合爆炸，如设备 x 渲染器 | 只有一个变化维度 |
| Composite | 用树结构统一处理个体和组合 | 菜单、组织、文件树、规则树 | 组合与叶子行为差异太大 |
| Decorator | 动态叠加职责，保持原接口 | 日志、缓存、权限、重试、压缩等可组合增强 | 增强顺序不清楚或调试成本过高 |
| Facade | 提供子系统简化入口 | 隐藏复杂模块、稳定外部 API | 变成万能服务或吞掉领域语义 |
| Flyweight | 共享细粒度不可变对象以省内存 | 大量重复值对象、字符/样式/规则元数据 | 对象有可变状态或数量不大 |
| Proxy | 控制对目标对象访问 | 远程、懒加载、权限、缓存、事务边界 | 只为“看起来高级”包一层 |

## 行为型模式

| 模式 | 意图 | 适用 | 避免 |
| :--- | :--- | :--- | :--- |
| Chain of Responsibility | 请求沿处理链传递，处理者可选择处理或转交 | 校验链、过滤器、中间件、审批链 | 处理顺序和失败语义不明确 |
| Command | 将请求封装为对象/消息 | 撤销、队列、审计、重试、异步任务 | 只是一次普通函数调用 |
| Interpreter | 为小语言定义语法和解释器 | 规则表达式、查询 DSL、简单脚本 | 语法复杂到需要 parser/编译器工具 |
| Iterator | 顺序访问集合而不暴露内部结构 | 自定义集合、流式读取、分页游标 | 语言内建迭代已足够 |
| Mediator | 用中介协调多个对象交互 | UI 控件协作、模块间复杂网状依赖 | 中介变成全知中心 |
| Memento | 捕获对象状态以便恢复 | 撤销、快照、草稿恢复 | 状态巨大或包含敏感资源 |
| Observer | 一对多通知依赖 | 事件通知、UI 更新、领域事件发布 | 需要强一致事务内同步结果 |
| State | 把状态相关行为封装到状态对象 | 状态机行为差异大，转换规则清楚 | 状态少且逻辑简单 |
| Strategy | 封装可替换算法 | 定价、排序、路由、风控策略 | 只有一个算法且不会变化 |
| Template Method | 父流程固定，子步骤可变 | 流程骨架稳定、步骤实现差异 | 组合/高阶函数更直接 |
| Visitor | 给对象结构添加新操作而不改结构类 | AST、报表导出、多操作作用于稳定结构 | 对象结构频繁变化 |

## 选择顺序

1. 创建对象复杂或产品族变化：优先看创建型。
2. 接口不兼容、对象组合或访问控制复杂：优先看结构型。
3. 算法、状态、请求、通知或对象协作变化：优先看行为型。
4. 同时命中多类时，只保留一个主模式；辅助模式必须有独立职责。

## 质量门槛

- 模式必须绑定一个明确变化轴。
- 角色命名要来自业务或技术边界，不要只叫 `ConcreteXxx`。
- 模式不得隐藏错误处理、事务边界、权限边界和生命周期。
- 能用语言惯用机制清晰表达时，不强行制造类层级。
