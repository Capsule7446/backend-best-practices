# Application 评分规则（workflow 门禁专用）

> 本文件由 **workflow 门禁**持有和消费；`ddd-application-review` 只产出证据表与发现，不打分。
> 评分两阶段：**硬门禁优先，客观覆盖率计分其次**——高分不能抵消事务、授权、幂等或分层错误。

## 1. 硬门禁（任一命中直接 fail）

| # | 硬门禁 |
| :-- | :-- |
| H1 | 关键用例没有唯一 Application Handler / Input Port |
| H2 | Controller、Consumer 或 Job 直接调用 Repository 或修改聚合 |
| H3 | Domain 依赖 Application / Infrastructure |
| H4 | Application 直接依赖 ORM、数据库、消息或第三方具体 SDK |
| H5 | 关键不变量只在 Application 检查，绕开 Handler 即可产生非法状态 |
| H6 | 写用例没有事务/UoW 和回滚语义 |
| H7 | 跨多个聚合做未解释的单事务写 |
| H8 | 要求可靠交付，但存在"数据库已提交、事件永久丢失"窗口 |
| H9 | 可重试命令或至少一次消息存在重复副作用，但没有幂等/去重 |
| H10 | 关键用例缺授权或存在跨租户路径 |
| H11 | 长事务包含无界网络调用、无限重试或重试业务拒绝 |
| H12 | 关键用例或适用的事务、授权、并发、幂等、事件测试缺失或失败（实现态） |
| H13 | 关键长流程进程崩溃后无法恢复、补偿或人工介入 |

## 2. 维度权重

| 维度 | 权重 |
| :-- | --: |
| D1 用例完整性与可追踪性 | 15 |
| D2 分层职责与依赖方向 | 15 |
| D3 编排清晰度与领域委派 | 15 |
| D4 事务、并发与一致性 | 15 |
| D5 端口、契约与错误语义 | 10 |
| D6 可靠性、幂等与事件交付 | 10 |
| D7 授权、租户与数据边界 | 10 |
| D8 测试与可执行证据 | 10 |

计分：`维度覆盖率 = 满足项 / 适用项`（取审查证据表的分子分母）；`总分 = Σ(权重 × 覆盖率)`。`N/A` 不进分母且必须有理由；关键指标不得标 `N/A`。不使用行覆盖率作为主要评分。

## 3. 判定

三个判定桶按顺序判且**穷尽**（不满足上一桶即落入下一桶，不存在无归属区间）：

| 判定 | 条件（按序判定）| 允许动作 |
| :-- | :-- | :-- |
| **pass** | 无硬门禁；总分 ≥ 85；D1/D2/D4/D7/D8 各 ≥ 90%；其余 ≥ 75%；关键用例覆盖率 100%；无 critical/high 发现 | 放行进入下一阶段/生产 |
| **conditional_pass** | 未达 pass，且：无硬门禁；总分 ≥ 70；核心维度（D1/D2/D4/D7/D8）各 ≥ 75%；关键覆盖率 100%；每个未达项有修复动作与复审标准 | 仅可继续规范/骨架/非生产实现，**不得生产切流** |
| **fail** | 其余一切情形（命中任一硬门禁；或未达 conditional_pass 下限；或分数无分母/证据不可审计）| 按回溯矩阵重跑上游 |

## 4. 设计态与实现态

- **设计态**（用例+编排评审后）：D8 按设计声明的测试义务完整性计；允许 `conditional_pass`。
- **实现态**（实现+测试完成后）：D8 按可执行证据计（test_file+test_name+status）；生产交付前必须 **pass**。
